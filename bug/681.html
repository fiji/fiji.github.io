Mats, just for the record: it is always good if you can provide a small macro demonstrating your problem so that developers (who are chronically overworked, unless they're retired ;-)) can reproduce the problem quickly.

Careful examination of the issue shows that the macro actually runs fine in Fiji *as long* as the sample images are not cached via File>Open Samples>Cache Sample Images.

This, in turn, suggests that there is an inconsistency between local and remote files, and indeed, painstakingly single-stepping a debugger through the relevant parts of the code turns up the real issue: the DICOM class expects the InputStream of the DICOM file to be open, still, when it is not:

https://github.com/fiji/imagej1/blob/8e330d85c341d0e4cb685b0559b26b08c67f836e/ij/plugin/DICOM.java#L92

Noting that the difference between the success and the failure described by Wayne basically boiling down to the DICOM being opened as a file in the unsuccessful case and as a URL in the other case, this hints at a difference in handling between URLs and files. And indeed, even in a Fiji where the sample images have been cached, a simple modification to the macro makes it work again!

-- snip --
run("URL...", "url=file://localhost/Applications/Fiji.app/samples/ct.dcm.zip");
key="0008,0060"; print(key+" = "+getInfo(key));
key="0028,1050"; print(key+" = "+getInfo(key));
key="0028,1051"; print(key+" = "+getInfo(key));
key="0028,1052"; print(key+" = "+getInfo(key));
-- snap --

(If you want to make this work on your machine, you might have to adjust the path in the first line, of course, if your Fiji does not live in /Applications/Fiji.app/, to point to the correct location instead.)

And indeed, if you modify the macro again so it looks like this:

-- snip --
open("/Applications/Fiji.app/samples/ct.dcm.zip");
key="0008,0060"; print(key+" = "+getInfo(key));
key="0028,1050"; print(key+" = "+getInfo(key));
key="0028,1051"; print(key+" = "+getInfo(key));
key="0028,1052"; print(key+" = "+getInfo(key));
-- snap --

(again, of course, adjusting the path in the first line to reflect where Fiji lives on your computer) and run it with bare-bones ImageJ 1.x, e.g. by calling

    java -jar /Applications/Fiji.spp/jars/ij-1.48i.jar dicom.ijm

(of course, again adjusting the path to point to your local Fiji's ij.jar, and also assuming that you saved the macro under the name "dicom.ijm" in the current directory), ImageJ 1.x fails just like Wayne described Fiji's failure to show the correct values.

This actually suggests that it works with the sample image only due to a bug: the input stream which opened the CT sample was never closed. (And only because it was loaded via URL rather than directly from disk.)

Now, digging even deeper into the issue, it turns out that the breakage happened between ImageJ 1.48g and ImageJ 1.48h, here is the (huge) diff:

https://github.com/fiji/ImageJA/commit/b220cf6041e96c73d63487113791eb5e120b9eda

And since nothing in that commit message gives a clue about any change that could relate to DICOM, and since that commit indeed does not even *change* DICOM.java, I again fired up the debugger and went through that incredibly exciting process of single-stepping through a mind-numbing number of lines, only to find out that the culprit is actually the change at

https://github.com/fiji/ImageJA/commit/b220cf6041e96c73d63487113791eb5e120b9eda#diff-efe686b3fe15ccbc63e143569d631309L1436

Turns out that ij/macro/Functions.java was changed not to special-case getInfo() with DICOM-style "0123.4567" keys anymore, and instead letting the DICOM plugin try to parse the file again, thanks to the DICOM class extending ImagePlus and overriding the getInfo() method. Except that that method requires the input stream from which the image was read to be open still (and according to the source code at https://github.com/fiji/ImageJA/blob/b220cf6041e96c73d63487113791eb5e120b9eda/ij/plugin/DICOM.java#L489 the file itself must be at most 400 kilobyte large for this to work).

Note: it is considered good practice to close input streams when you are done with them because there is actually a limit how many files can be kept open at the same time. So: Wayne's macro works with the sample image *only* because there is a bug in the code that opens DICOM images via URLs!

Unfortunately, there are no good commit messages in the more fine-grained source code repository of ImageJ 1.x' daily builds so that https://github.com/fiji/imagej1/commit/16436c5d5e378838155ff60fb9aa67380eff2335 (which is the change that broke things, to be precise, it was this change: https://github.com/fiji/imagej1/commit/16436c5d5e378838155ff60fb9aa67380eff2335#diff-efe686b3fe15ccbc63e143569d631309L1436) does not provide me with an explanation of the rationale behind the change.

Another thing to note is that the output of the working macro (i.e. with the URL rather than the file path) changed from

0028,1051 =  500 
0028,1052 =   -1024

to

0008,0060 =  Modality: CT
0028,1050 =  Window Center: 50

which might be perceived as an improvement to users, but undoubtedly presents a regression for macros that worked quite fine with older ImageJ 1.x versions.
